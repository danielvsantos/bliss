# 15. Insights API

This document specifies the `/api/insights` endpoint exposed by the Next.js API layer. The endpoint surfaces tiered financial insights generated by the backend service (see `docs/specs/backend/15-insights-engine.md` for the generation pipeline), handles dismiss/restore state, and dispatches on-demand generation requests to the backend via a fire-and-forget pattern.

## 15.1. Overview

The API layer does not generate insights. It serves them from the `Insight` table with tenant isolation, filtering, and aggregate summaries, and it brokers three operations:

| Method | Route           | Purpose                                                                  |
|--------|-----------------|--------------------------------------------------------------------------|
| GET    | `/api/insights` | List insights with filters; also returns per-tier and per-category summaries |
| PUT    | `/api/insights` | Dismiss or restore a single insight (ownership-checked)                  |
| POST   | `/api/insights` | Fire-and-forget on-demand generation request (any tier)                  |

- **File**: `pages/api/insights.js`
- **Authentication**: JWT via `withAuth` — all operations are tenant-scoped by `user.tenantId`
- **Rate limiting**: `rateLimiters.accounts`
- **CORS**: `cors(req, res)` — returns early on preflight

Multi-tenant isolation is enforced at the query level — every Prisma query includes `tenantId: user.tenantId`. The API never trusts a client-supplied `tenantId`.

## 15.2. Validation Constants

Two server-side whitelists gate filter inputs. Values outside the whitelist are silently dropped (not rejected) on GET so the endpoint stays forgiving to typos from query strings, while the POST endpoint rejects missing or invalid tier values with a 400.

```javascript
const VALID_TIERS = ['MONTHLY', 'QUARTERLY', 'ANNUAL', 'PORTFOLIO'];
const VALID_CATEGORIES = ['SPENDING', 'INCOME', 'SAVINGS', 'PORTFOLIO', 'DEBT', 'NET_WORTH'];
```

These constants mirror the backend `insightService.js` exports and the frontend `InsightTier` / `InsightCategory` type unions. `DAILY` is **not** a member — the retired tier is explicitly rejected by POST with a 400.

## 15.3. `GET /api/insights`

Lists insights for the authenticated tenant.

### 15.3.1. Query Parameters

| Parameter          | Type      | Default  | Description                                                  |
|--------------------|-----------|----------|--------------------------------------------------------------|
| `limit`            | `int`     | `20`     | Page size                                                    |
| `offset`           | `int`     | `0`      | Page offset                                                  |
| `lens`             | `string`  | —        | Filter by single lens (e.g. `SPENDING_VELOCITY`)             |
| `severity`         | `string`  | —        | Filter by severity (`POSITIVE`, `INFO`, `WARNING`, `CRITICAL`) |
| `tier`             | `string`  | —        | Filter by tier. Silently dropped if not in `VALID_TIERS`     |
| `category`         | `string`  | —        | Filter by category. Silently dropped if not in `VALID_CATEGORIES` |
| `periodKey`        | `string`  | —        | Filter by exact period key (see backend spec 15.3.3)         |
| `includeDismissed` | `boolean` | `false`  | When `"true"`, includes dismissed insights                   |
| `latestOnly`       | `boolean` | `false`  | Reserved — currently only influences the default order       |

The default order is `[{ priority: 'desc' }, { createdAt: 'desc' }]`.

### 15.3.2. Response

```json
{
  "insights": [
    {
      "id": "cuid",
      "lens": "SPENDING_VELOCITY",
      "tier": "MONTHLY",
      "category": "SPENDING",
      "periodKey": "2026-03",
      "severity": "WARNING",
      "title": "Dining spend up 42% vs February",
      "body": "…",
      "priority": 72,
      "metadata": {
        "actionTypes": ["BUDGET_OPTIMIZATION"],
        "relatedLenses": ["SAVINGS_RATE"],
        "suggestedAction": "Consider a monthly dining budget"
      },
      "dataHash": "sha256-hex",
      "dismissed": false,
      "expiresAt": "2028-04-02T06:05:00.000Z",
      "createdAt": "2026-04-02T06:05:00.000Z"
    }
  ],
  "total": 42,
  "tierSummary": {
    "MONTHLY":   { "latestDate": "2026-03-31", "latestCreatedAt": "2026-04-02T06:05:00.000Z" },
    "QUARTERLY": { "latestDate": "2026-03-31", "latestCreatedAt": "2026-04-03T06:05:00.000Z" }
  },
  "categoryCounts": {
    "SPENDING": 12,
    "PORTFOLIO": 8,
    "NET_WORTH": 3
  }
}
```

Key notes:

- **`tierSummary`** is produced via `prisma.insight.groupBy({ by: ['tier'], _max: { date, createdAt } })` over **all** tenant insights (no filters applied). It powers the frontend "last updated" label per tier.
- **`categoryCounts`** is produced via `prisma.insight.groupBy({ by: ['category'], _count: { id } })` with only the `dismissed` filter applied. It drives the category tab badges in the frontend.
- The legacy `latestBatchDate` field is **not** returned — callers must use `tierSummary[tier].latestCreatedAt` instead.

## 15.4. `PUT /api/insights`

Dismisses or restores a single insight.

### 15.4.1. Request Body

```json
{ "insightId": "cuid", "dismissed": true }
```

Both fields are required. `dismissed` must be a boolean (not `"true"` / `"false"` strings).

### 15.4.2. Responses

| Status | Condition                                                                 |
|--------|---------------------------------------------------------------------------|
| 200    | Updated insight object                                                    |
| 400    | `insightId` missing OR `dismissed` not a boolean                          |
| 404    | Insight not found OR belongs to a different tenant                        |
| 500    | Server error                                                              |

The ownership check runs before the update:

```javascript
const insight = await prisma.insight.findFirst({
  where: { id: insightId, tenantId: user.tenantId },
});
if (!insight) return res.status(404).json({ error: 'Insight not found' });
```

## 15.5. `POST /api/insights`

Triggers on-demand insight generation for the authenticated tenant. The request is fire-and-forget: the API layer POSTs to the backend's internal `/api/insights/generate` endpoint and returns 202 without waiting for the result.

### 15.5.1. Request Body

| Field       | Type      | Required | Notes                                                              |
|-------------|-----------|----------|--------------------------------------------------------------------|
| `tier`      | `string`  | ✅       | Must be in `VALID_TIERS`. No default — the DAILY fallback was retired in v1.1 |
| `year`      | `int`     |          | Required by backend for `MONTHLY` / `QUARTERLY` / `ANNUAL` tiers   |
| `month`     | `int`     |          | Required by backend for `MONTHLY`                                  |
| `quarter`   | `int`     |          | Required by backend for `QUARTERLY`                                |
| `periodKey` | `string`  |          | Optional — backend computes from year/month/quarter if omitted     |
| `force`     | `boolean` |          | Bypass completeness gate + dedup. Also accepts `"true"` string     |

Per-tier required params (year/month/quarter) are **not** validated at the API layer — the backend route enforces them (see `docs/specs/backend/15-insights-engine.md` section 15.9.1 and the backend `POST /api/insights/generate` route). The API layer validates that `tier` is present and a member of `VALID_TIERS`.

### 15.5.2. Responses

| Status | Condition                                                         |
|--------|-------------------------------------------------------------------|
| 202    | `{ message, tier }` — dispatched                                  |
| 400    | `tier` missing OR `tier` not in `VALID_TIERS` (includes `DAILY`)  |
| 401    | Unauthorized                                                      |
| 500    | Server error                                                      |

### 15.5.3. Backend Dispatch

```javascript
fetch(`${BACKEND_URL}/api/insights/generate`, {
  method: 'POST',
  headers: {
    'Content-Type': 'application/json',
    'X-API-KEY': BACKEND_API_KEY,
  },
  body: JSON.stringify({
    tenantId: user.tenantId,
    tier,
    year: year ? parseInt(year, 10) : undefined,
    month: month ? parseInt(month, 10) : undefined,
    quarter: quarter ? parseInt(quarter, 10) : undefined,
    periodKey,
    force: force === true || force === 'true',
  }),
}).catch((err) => Sentry.captureException(err));
```

The fetch is intentionally not awaited. Failures are logged to Sentry but don't affect the 202 response — the backend will retry via BullMQ if the enqueue itself fails. This matches the fire-and-forget pattern established for other long-running jobs.

## 15.6. Method Validation

Unsupported methods return 405 with the `Allow: GET, PUT, POST` header.

## 15.7. Tests

- **File**: `__tests__/unit/api/insights.test.ts`
  - GET: default filter (excludes dismissed), lens filter, tier/category validation (accepted + silently dropped), periodKey filter, combined filters, `includeDismissed=true`, custom limit/offset
  - GET: `tierSummary` aggregation from `insight.groupBy` results
  - GET: `categoryCounts` reduction into flat map
  - PUT: dismiss/restore happy path, missing `insightId`, non-boolean `dismissed`, ownership check (404)
  - POST: missing tier → 400, `tier=DAILY` → 400 (retired tier rejected), tiered params (MONTHLY/QUARTERLY/ANNUAL/PORTFOLIO), `force` passthrough, invalid tier rejection
  - 405 on unsupported methods

Test pattern: mocked-handler approach (`vi.mock` for `withAuth`, `prisma`, `rateLimit`, `cors`, `@sentry/nextjs`, and global `fetch`).
